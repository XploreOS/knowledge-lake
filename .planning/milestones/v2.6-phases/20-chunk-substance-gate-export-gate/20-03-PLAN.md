<objective>
Gate the gold RAG corpus export on chunk-level `substance_passed` (from Plan 20-01's gate) instead of the document-level `enriched.quality_score`, and add a `version` tag to newly-generated eval/instruction dataset examples derived from `filter_config_version`.

Purpose: `enriched.quality_score` is identical for every chunk of the same document and cannot distinguish within-document variance (a document with both a clinical table and a cookie-banner section currently exports both to gold). This is the mechanism behind the milestone's "<2% junk in gold" success criterion (EXPORT-01). `version` tagging (EXPORT-02) prevents a v2.6 quality improvement from being silently scored against a pre-v2.6, boilerplate-grounded eval set.

Output: `export_rag_corpus()` filters on chunk-level `substance_passed`; `generate_qa_example()`/`generate_instruction_example()` tag their output with `version`.
</objective>

<operational_followup>
## Post-Phase Operator Action: Regenerating Eval Datasets (EXPORT-02)

This plan tags NEWLY-generated `dataset_examples` rows with `version` — it does not regenerate, migrate, or delete pre-v2.6 rows grounded in boilerplate (D-2 forward-only, per the `prohibitions` block above). REQUIREMENTS.md's EXPORT-02 acceptance text ("Regeneration after v2.6 produces sets grounded in clinical content, not 'Featured'") is satisfied as a **documented operator action**, using tooling this plan does not need to build because it already exists:

1. Re-process the sources whose eval examples are stale (`klake process <source_id>`, or trigger the Dagster `chunk_document`/`enrich_document` assets for that source). Plan 20-01's PIPE-01 fix means chunk artifacts get fresh `content_hash` values under the current `settings.chunk_quality.filter_config_version`, so this step naturally produces gate-improved chunks.
2. For each chunk or enriched_document artifact that should back an eval example, run the existing CLI command: `klake generate-dataset qa <chunk_artifact_id> --dataset-name <name>` (or `klake generate-dataset instruction <enriched_document_artifact_id> --dataset-name <name>`). This calls `pipeline.datasets.generate_qa_example`/`generate_instruction_example` — the exact functions Task 2 above tags with `version` — against the newly re-processed artifact's `content_hash`, so it is a cache MISS against `_dataset_gen_cache_key()` and produces a fresh, version-tagged `DatasetExample` grounded in the new gate's output.

No new code task is required to satisfy this: `klake generate-dataset` (`src/knowledge_lake/cli/app.py::cmd_generate_dataset`) is unmodified by this plan and is already the regeneration mechanism.
</operational_followup>
