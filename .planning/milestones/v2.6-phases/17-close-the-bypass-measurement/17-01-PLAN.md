<objective>
Retrofit `pipeline/clean.py::clean()` so it can (a) accept an in-memory `ParsedDoc` and clean it at section granularity without dropping sections, (b) compute a parent-scoped content hash that cannot collide across documents, and (c) unconditionally record kept/rejected/considered counts with a runtime conservation check. This is the highest-risk, foundational plan of Phase 17 — every other plan (17-02 Dagster wiring, 17-03 CLI wiring, 17-04 quality-audit) depends on the new `clean()` signature and return shape this plan produces.

Purpose: Per CONTEXT.md D-01/D-04/D-09/D-11 and RESEARCH.md Pitfall 1, the CLEAN-03 hash fix and the CLEAN-01/02 section-forwarding substrate must ship together — activating the forwarding without the hash fix would make a dormant lineage-corruption bug live. This plan closes both in one wave, before any downstream consumer (Dagster asset, CLI, quality-audit) starts reading `clean()`'s new `cleaned_doc` output. Per D-12, Task 2 wires the conservation invariant at the ONLY granularity available today — per-section within a single `clean()` call — since no section classifier exists yet; Phase 19 will populate this same infrastructure with richer section-level rejection reasons once the classifier lands, without needing to change this plan's mechanism.
Output: `clean()` with a new `parsed_doc` kwarg, WR-05 parent-scoped hashing, a `cleaned_doc` return value, and QUAL-04/QUAL-05 rejection/conservation metadata — all backward-compatible with existing callers that don't pass `parsed_doc`.
</objective>

- **CLEAN-01 (unclassified):** When `parsed_doc.sections` is empty (a parser that legitimately
  returns zero sections) reaches `clean_document`, this plan assumes `clean()` still succeeds and
  returns a `cleaned_doc` with `sections=[]` — matching `chunk.py`'s existing "no sections: emit
  full text as single chunk" fallback (`chunk.py:204-215`). This is NOT itself a QUAL-05
  conservation violation (0 rejected + 0 kept == 0 considered holds trivially) — the zero-sections
  case is logged as `clean.zero_sections`, not raised. Not covered by a dedicated automated test in
  this plan; Task 2's zero-section test covers the mechanism but not the full asset-graph path
  (that lives in Plan 17-02, which also does not add a dedicated empty-sections integration test).
- **CLEAN-03 (unclassified):** When `cleaned_text` is the empty string (a document that reduces to
  nothing after boilerplate stripping) for two different `parsed_artifact_id`s, this plan assumes
  the two artifacts still get distinct `content_hash` values, because the hash input is
  `f"{parsed_artifact_id}:{cleaned_text}"` — the parent-ID prefix alone guarantees uniqueness even
  when `cleaned_text == ""`. This is implicit in the WR-05 hash design but not asserted by a
  dedicated empty-cleaned-text test in Task 1's test list; non-blocking, flagged for the executor
  to add if time allows.
