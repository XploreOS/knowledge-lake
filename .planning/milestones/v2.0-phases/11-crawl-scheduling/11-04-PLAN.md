<objective>
Expose the two operator paths for setting a crawl schedule (D-05): (a) a `crawl_schedule:` key in `domains/*/sources.yaml` persisted at `domain-init`, and (b) a `klake set-schedule` CLI verb to set/clear a schedule on an existing source. Both validate the cron with Dagster's `is_valid_cron_string` before any write, so a bad cron is rejected at write time rather than silently at sensor-tick time.

Purpose: Scheduling is opt-in and operator-controlled; the sensor only ever sees validated cron strings.
Output: `SourceEntry.crawl_schedule`, schedule persistence in `cmd_init`, and the `set-schedule` command.

Decisions implemented: D-03 (5-field UTC cron string validated with is_valid_cron_string), D-05 (schedule set two ways — sources.yaml at domain-init and a CLI set/clear verb; column, not config).
</objective>
