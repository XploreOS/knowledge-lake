# Quick Task 260715-51d: KL-01 domain filtering + KL-02 LLM pricing

## Goal

Close two high-severity findings from `.planning/E2E-GAP-ANALYSIS.md`:

- **KL-01** — `domain=` must actually filter export rows, not just name the S3 path.
- **KL-02** — LLM cost tracking must reflect real spend so `budget_usd` means what it says.

## Locked decision

**`domain=` filters.** The user resolved the KL-01 ambiguity explicitly: the
parameter selects rows, and the path/tag then truthfully describes the contents.
Do not implement the "rename to `path_label`" alternative.

---

## KL-01 — investigation (established; do not re-derive)

All three export functions accept `domain` and use it **only** at the write step
to build the key and the S3 tag:

```python
domain_seg = domain or _UNCLASSIFIED_DOMAIN
key = f"{s.export.gold_prefix}/{domain_seg}/rag_corpus/{export_id}.parquet"
storage.put_object(key, buf.getvalue(), tags={"domain": domain_seg, ...})
```

Row queries ignore it entirely:

| Function | Line | Query | Domain reachable via |
|---|---|---|---|
| `export_rag_corpus` | 277 | `list_artifacts_by_type(session, "chunk")` | `chunk.source_id` (already resolved per-row as `row_domain`) |
| `export_pretrain_corpus` | 392 | `list_artifacts_by_type(session, "curated_document")` | `curated.source_id` |
| `export_finetune_dataset` | 494 | `list_dataset_examples(session, dataset.id)` | `artifact.source_id` of each example's source artifact |

Helper already exists: `registry_repo.get_domain_for_source(session, source_id) -> str | None`
(reads `Source.config["domain"]`; returns None if absent). It uses
`session.get(Source, …)`, so SQLAlchemy's identity map makes the per-row lookup
cheap within one session — no need to pre-build a map, but do not add an N+1
across sessions.

**Proven impact (from the audit):** `export_rag_corpus(domain="aviation")` wrote
`gold/aviation/rag_corpus/….parquet` tagged `domain=aviation` containing 51
aviation / 53 functional-medicine / 29 null rows — 62% foreign.

### Semantics to implement

- `domain="aviation"` → include **only** rows whose source domain == `"aviation"`
  (strict equality; rows with `domain=None` are NOT included). Path/tag: `aviation`.
- `domain=None` → **no filtering** (all domains), path/tag `_unclassified` as today.
  This preserves the current CLI/API behavior exactly — do not change it.

> Known wart, explicitly OUT OF SCOPE: with `domain=None` the `_unclassified`
> path segment and S3 tag still describe an all-domain export. Renaming that
> segment would change the STORE-03 gold-zone layout. Leave it; note it in the
> SUMMARY as a follow-up. The reported bug — `domain=X` not filtering — is what
> this task fixes.

## KL-02 — investigation (established; do not re-derive)

`bootstrap_llm_pricing()` registers prices keyed by **Bedrock model ID**
(`bedrock/us.anthropic.claude-haiku-4-5…`), but every call goes through the proxy
and comes back with `response.model == "cheap_model"` (the alias). The keys never
match → `completion_cost()` raises on every call → `compute_call_cost()` silently
returns a flat estimate that under-prices by **2.4× (cheap)** to **9× (strong)**,
making `budget_usd=5.0` behave like $12–$45. `eval_model` has no registered price
at all (only `cheap_model_bedrock_id` / `strong_model_bedrock_id` exist).

### Probe result — this changes the fix

A **live** proxy call was measured:

```
response.model                      : cheap_model
_hidden_params["response_cost"]     : 3.74e-05     ← proxy already computed it, correctly
completion_cost() (alias registered): 2.72e-05     ← from the hand-maintained settings table
```

Two conclusions:

1. **The LiteLLM proxy already returns the correct cost** in
   `response._hidden_params["response_cost"]`, computed from the real backend
   model's pricing. We have been ignoring it and hand-maintaining a table that
   drifts. This is the authoritative source and needs no per-alias maintenance —
   it covers `eval_model` and any future alias for free.
2. Registering under the **alias** name does make `completion_cost()` resolve
   (the two figures differ only because the settings table's numbers are
   approximate) — so it is a viable *fallback*, not the primary.

### Fix design (ordered fallback chain)

Rewrite `compute_call_cost()` to try, in order:

1. `response._hidden_params["response_cost"]` — use when present and `> 0`. Authoritative.
2. `litellm.completion_cost(completion_response=response)` — works once
   `bootstrap_llm_pricing` registers under alias names.
3. Flat estimate from `fallback_cost_per_1k_*` — last resort, and log at
   **warning** with a message that says cost is being under-estimated, not the
   current bare `enrich.cost_calc_failed`.

And change `bootstrap_llm_pricing()` to register under the **alias names**
(`cheap_model`, `strong_model`, `eval_model`) with `litellm_provider: "openai"`
(the wire protocol the proxy speaks), instead of the Bedrock IDs. Add
`eval_model` price settings mirroring the existing `strong_model` ones
(`eval_model` maps to the same Sonnet model in `infra/litellm/config.yaml`).

Keep the never-raise contract: pricing must never break enrichment (D-05).

- **files:** `src/knowledge_lake/pipeline/export.py`
- **action:** In `export_rag_corpus`, `export_pretrain_corpus` and
