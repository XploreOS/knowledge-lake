### Task 1 — KL-07: SSRF guard misses reserved ranges

- **files:** `src/knowledge_lake/pipeline/ingest.py`
- **background:** `validate_public_url()` is otherwise solid (https-only,
  `getaddrinfo()` over all A/AAAA, IPv4-mapped unwrapping, redirect
  re-validation). Its hand-rolled `_PRIVATE_NETS` list (9 networks) misses:
  `0.0.0.0/8` (**reaches localhost on Linux**), `100.64.0.0/10` CGNAT,
  `198.18.0.0/15`, `192.0.0.0/24`, `240.0.0.0/4`, `::`. Measured — stdlib
  `addr.is_global` returns False for **every** missed range, and True for the
  legitimately public ones.
- **action:** Replace the `_PRIVATE_NETS` membership test with a
  `not addr.is_global` check (after the existing IPv4-mapped unwrap). Keep
  `_PRIVATE_NETS` as a documented list of the *notable* blocked ranges, and keep
  the error message specific about what was blocked and why. Reachability
  matters: `klake discover` auto-registers URLs harvested from SearXNG.
- **verify:** Controls still blocked (`10.0.0.1`, `169.254.169.254`, `127.0.0.1`);
  newly blocked (`0.0.0.0`, `100.64.0.1`, `198.18.0.1`, `192.0.0.1`, `240.0.0.1`, `::`);
  a real public host (e.g. `www.faa.gov`) still ALLOWED — do not over-block.
- **done:** Reserved ranges rejected; existing SSRF tests pass.

> Known limitation to leave alone: validation resolves the hostname, then the
> HTTP client resolves again on connect — a DNS-rebinding TOCTOU. Pinning the
> validated IP is a separate change; note it, don't attempt it here.

### Task 2 — KL-11: wiki summaries cut mid-sentence

- **files:** `src/knowledge_lake/pipeline/wiki.py`
- **background:** `wiki.py:538` — `lead = (doc["summary"] or "")[:500]`. A hard
  character slice: no word boundary, no ellipsis. The stored summary is fine; the
  published page is what breaks. Real output ended `"...as their frame of
  reference rather than"` while the stored text continued
  `"...rather than height above ground or groundspeed."`
- **action:** Truncate on a boundary and signal it. Prefer the last sentence end
  within `summary_excerpt_chars`; else the last word boundary; append an ellipsis
  when truncation occurred. Never emit a mid-word cut. Do not change
  `summary_excerpt_chars` (500) — the limit is fine, the slicing isn't.
- **verify:** A >500-char summary renders ending at a boundary + ellipsis; a
  <500-char summary is emitted unchanged with **no** ellipsis.
- **done:** No mid-sentence truncation in wiki output.

## Wave 2 — KL-04 / KL-05 / KL-06 (one design change)

**Locked decision.** These three findings are one problem. D-01 made
`enrich_document` and `curate_document_asset` "parallel branches that do not
block" — but the dependencies are **real**, not optional:

- `curate` reads the enriched sibling for the 40% enrich term of its composite
  (`compute_composite_quality_score`: `parse*0.30 + enrich*0.40 + filters*0.30`),
  and silently substitutes **0.5** when enrichment hasn't run yet.
- `index` reads the enriched sibling at runtime for the payload.

Measured consequence on the real aviation document (`parse=0.989`,
`filter_pass_ratio=1.0`):

| curate ran | enrich term | composite |
|---|---|---|
| before enrich | 0.5 (default) | **0.797** ← what was actually observed |
| after enrich | 0.92 (real) | **0.965** |

**21% swing on the same document, from scheduling order alone.** That is false
parallelism producing nondeterministic scores.

### Task 4 — Make the chain explicit + fix payload semantics

- **files:** `src/knowledge_lake/dagster_defs/assets.py`,
  `src/knowledge_lake/pipeline/index.py`, `src/knowledge_lake/registry/repo.py`,
  `src/knowledge_lake/cli/app.py` (+ API if it exposes reindex)
- **action:**
  1. **Ordering (KL-06).** Use Dagster non-data deps (`@asset(deps=[...])` —
     confirmed available in 1.13.11) to enforce
     `clean → enrich → curate → chunk → embed → index`:
     - `curate_document_asset`: add `deps=[enrich_document]`
     - `chunk_document`: add `deps=[curate_document_asset]`
     Keep the existing **data** inputs as they are (both still take
     `clean_document`, preserving the in-memory `ParsedDoc` forwarding — do not
     re-plumb data flow). `index_chunks` then depends on curate transitively.
     Update the D-01 docstrings that currently assert "does not block" — they
     will be actively wrong.
  2. **Payload (KL-04/KL-05).** Add
     `registry_repo.get_curated_artifact_for_parsed(session, parsed_artifact_id)`
     mirroring the existing `get_enriched_artifact_for_parsed` (walks
     parsed → cleaned → curated). In `pipeline/index.py`, resolve
     `quality_score = curated.quality_score ?? enriched.quality_score ?? None`.
     Keep `document_type` / `keywords` / `title` sourced from enrichment metadata
     as today. Document precedence in the payload docstring.
  3. **Repair path (KL-06).** `reindex` currently copies points verbatim, so a
     chunk indexed pre-enrichment can never be repaired without full re-ingest.
     Add an opt-in payload refresh (e.g. `klake index --refresh-payload`) that
     re-derives the payload from the registry per point instead of copying.
     Default OFF — preserve today's copy behavior.
- **verify:** Task 5 tests; plus a real check that a curated doc's composite
  reaches the search payload.
- **done:** Same document scores identically regardless of scheduling; curate's
  composite reaches search; stale payloads are repairable.

### Task 6 — KL-10: remove stale xfail markers, then enable xfail_strict

- **files:** `tests/**`, `pyproject.toml`
- **background:** At audit time 42 tests xpassed (35 unit + 7 integration) —
  markers like *"Wave 0 stub — implementation pending"* on features that now work
  (`test_domain_loader`, `test_cli_init_index`, `test_api_search_mode`,
  `test_healthcare_*`). With `xfail_strict` unset these can never fail the build:
  if the feature regressed, CI would record a tidy `xfail` and stay green.
- **action, in this order:**
  1. **Re-measure** the xpass set *after* Tasks 1-5 (`pytest -m "not browser" -rX`).
     Do NOT reuse the audit's list — Waves 1-2 may have changed it.
  2. Remove the `xfail` marker from every test that now **xpasses** (the feature
     works; the marker is a lie). Delete the marker and any now-unused import —
     do not convert to `skip`.
  3. **Leave markers on tests that genuinely `xfail`** (still failing, expected).
     `xfail_strict` only fails on XPASS, so real xfails are unaffected. Verify
     each remaining marker has a reason that is still true; if a marker's reason
     is stale but the test still fails, update the reason rather than deleting.
  4. Only once zero xpasses remain: set `xfail_strict = true` under
     `[tool.pytest.ini_options]` in `pyproject.toml`.
  5. Re-run everything. `xfail_strict` makes any future xpass a hard failure —
     that is the point.
- **verify:** `uv run pytest -m "not browser" -q` → **0 xpassed**, 0 failed,
  0 errors, with `xfail_strict = true` active.
- **done:** No test can silently pass-while-marked-failing again.

> If a test xpasses but the marker looks *deliberate* (e.g. a genuinely flaky
> or environment-dependent case rather than a "Wave 0 stub"), do NOT blindly
> strip it — report it in the SUMMARY and leave it, since `xfail_strict` would
> then fail the build. Flag the conflict rather than guessing.
