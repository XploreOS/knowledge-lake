## Phase Goal (slice framing)

**As a** knowledge-lake operator, **I want to** run DataTrove-style quality filters and a corpus-wide MinHash dedup pass over my cleaned corpus and get back a single composite quality score per document, **so that** I can see — via `klake curate`/`klake dedupe` or the API — exactly which documents are fit for downstream dataset generation and export, and which are near-duplicates or low quality, before I spend LLM budget or ship a gold-zone deliverable built on them.

<objective>
Implement CURATE-01 (DataTrove-style quality filters), CURATE-02 (corpus-wide MinHash dedup replacing Phase 3's transient per-call scan), and CURATE-03 (composite quality score, queryable via CLI/API) as a single vertical slice: `curated_document` artifacts parented on `cleaned_document` (D-01), a new `pipeline/curate.py` module, and CLI/API/Dagster wiring. Resolves D-04's DataTrove-inside-Dagster integration question — per RESEARCH.md's finding, this means calling DataTrove's real filter classes' `.filter(doc)` method directly against an in-memory `datatrove.data.Document` sourced from the existing Postgres+S3 registry, never `LocalPipelineExecutor`/`datatrove.io`'s file-based scaffolding, wrapped by a thin `@asset` exactly like every other pipeline stage.

Purpose: Turn Phase 3/4's cleaned + enriched corpus into a scored, dedup-flagged corpus that Plan 05-02 (dataset generation) and Plan 05-03 (export) can filter against.
Output: `src/knowledge_lake/pipeline/curate.py` (`curate_document()`, `batch_dedup_corpus()`), `quality/scorer.py::compute_composite_quality_score()`, `registry/repo.py` additions, `CurateSettings`, `klake curate`/`klake dedupe` CLI commands, `POST /curate` + `GET /curated-documents` API endpoints, `curate_document` Dagster asset.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cleaned document text -> DataTrove filter scoring | Untrusted crawled/uploaded document content flows into `GopherRepetitionFilter`/`GopherQualityFilter`/`C4QualityFilter` and MinHash signature computation |
| CLI/API artifact-ID inputs -> registry queries | User-supplied `cleaned_artifact_id`/`source_id`/`min_quality_score` flow into parameterized ORM queries |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Tampering | pipeline/curate.py score_document() | low | accept | Crawled document content could be crafted to game quality-filter heuristics (e.g. padding with repeated n-grams to pass Gopher thresholds); low severity because curation is annotate-only for MVP (CURATE-01 output is a queryable signal, not an auto-delete gate) and PROJECT.md restricts ingestion to public, non-adversarial regulatory sources |
