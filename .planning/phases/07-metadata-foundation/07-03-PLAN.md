## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| search() caller → Qdrant filter | source_name, format, source_id, tags are user-supplied filter strings passed as FieldCondition match values |
| API/CLI → search() | Plan 04 will validate these at the API boundary; at the search() level they are typed Optional[str]/list[str] |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-03-01 | Information Disclosure | ensure_payload_indexes — collection name | low | accept | collection_name is the physical collection string, already validated by _COLLECTION_NAME_RE guard in api/app.py before reaching this layer; internal test fixture uses a hardcoded string |
| T-07-03-02 | Denial of Service | search() tags list | medium | mitigate | tags is Optional[list[str]]; Plan 04 adds max_length=64 per-element constraint on the API Query param (RESEARCH.md security domain: "optional max_length on each element; follow existing max_length=64 pattern"). At the search() layer, list length is unbounded — acceptable because Qdrant handles arbitrary MatchAny lists safely; the API boundary is the enforcement point |
| T-07-03-03 | Injection | search() filter kwargs → Qdrant | low | accept | Qdrant parameterized FieldCondition/MatchValue/MatchAny are strongly-typed model objects; no string concatenation into query syntax. Qdrant client handles escaping internally. Path traversal note from RESEARCH.md security domain: moot for search filter strings |
</threat_model>
