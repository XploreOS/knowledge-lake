<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Pipeline caller → StorageBackend.put_raw/put_bronze | Domain value flows from caller (pipeline function) into key construction in s3.py; s3.py never calls the registry to resolve domain |
| StorageBackend.put_object → boto3 S3 client | Tags dict is URL-encoded and passed as Tagging= parameter; ClientError on tagging falls back to tagless write |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-01 | Denial of Service | _format_tags / S3 Tagging parameter | medium | mitigate | Tag value truncation to 256 chars in _format_tags(v[:256]) prevents InvalidTagValue ClientError from tag value overflow (D-09) |
| T-09-02 | Tampering | put_object best-effort fallback | low | accept | Tagging failure triggers tagless retry per D-10; object always lands; tags are convenience metadata only (registry is source of truth per D-07) |
| T-09-03 | Tampering | S3 key segment (domain from Source.config) | medium | accept | Domain values come from Source.config populated at register_source time; "_unclassified" fallback prevents None/empty key segments; S3 does not allow path-traversal exploitation via key names in the same way a filesystem does |
| T-09-SC | Tampering | npm/pip/cargo installs | low | accept | No new packages installed in this phase (RESEARCH.md: "No new packages required") |
</threat_model>
