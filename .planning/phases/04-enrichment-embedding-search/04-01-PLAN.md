Build the shared registry schema, settings, and repository-layer foundation that both the enrichment vertical slice (Plan 02) and the index/search vertical slice (Plan 03) depend on: map the existing quality_score column, add llm_spend and vector_collections tables via a new Alembic migration, add EnrichSettings/IndexSettings, and add every new repo.py function either later plan needs so neither plan has to touch registry/repo.py or registry/models.py again (avoiding same-wave file conflicts).

Output: registry/models.py with Artifact.quality_score mapped + LlmSpend + VectorCollection models; Alembic migration 0007; registry/repo.py with 7 new functions; config/settings.py with EnrichSettings + IndexSettings; ids.py with the enriched_document ID prefix.
</objective>

## Phase Goal (user story)

**As a** knowledge-lake operator, **I want to** have enrichment spend, prompt-cache, and vector-collection-alias state tracked in the registry, **so that** the enrichment and search slices built next can enforce budget caps and zero-downtime reindexing without inventing ad-hoc state.

> This is the Wave-1 schema/settings foundation for Phase 4 — it has no CLI/API surface of its own. Both the enrichment slice (04-02) and the index/search slice (04-03) require this registry schema and settings to exist first — a genuine dependency conflict (RESEARCH.md's llm_spend/vector_collections tables have no prior analog to reuse), not a difficulty-based split. Each of those two later plans delivers real operator-facing capability end-to-end (CLI + API + Dagster).
