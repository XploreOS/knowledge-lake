## User Constraints (from CONTEXT.md)

### Locked Decisions

**Substance Gate Wiring (QUAL-03)**
- D-01: The substance gate runs inside `chunk()` as a post-generation filter. After the raw chunk list is produced, `run_predicates()` (from `pipeline/quality/`, built in Phase 19) is applied to each chunk. Rejected chunks are excluded from the returned list (in enforce mode) or annotated but kept (in report mode). This ensures both Dagster and CLI paths (which both call `chunk()`) have automatic parity — no separate wiring needed.
- D-02: Each chunk's `metadata_` dict is annotated with `substance_passed: bool` and `rejection_reason: str | None` before the gate decision. These annotations persist through embed and index and are available to the export gate (EXPORT-01).
- D-03: `is_table=True` chunks are always exempt from the substance gate — they bypass all predicates and are marked `substance_passed=True`. This matches QUAL-03's acceptance criteria.
- D-04: The conservation invariant (QUAL-05, Phase 17) is checked at the end of `chunk()`: `rejected + kept == total_generated`. This catches broken predicates that silently drop chunks.

**FineWebQualityFilter at Chunk Scope (QUAL-02)**
- D-05: A `ChunkQualitySettings` Pydantic model is added to the settings hierarchy under `settings.chunk_quality`. It holds chunk-appropriate thresholds for `FineWebQualityFilter` (shorter text, different character distribution than document-level `CurateSettings`). Fields include `min_token_count`, `min_alpha_ratio`, `max_link_density`, `min_stopword_ratio`, and FineWebQualityFilter-specific overrides.
- D-06: `FineWebQualityFilter` is instantiated with chunk-scoped params from `ChunkQualitySettings`, not reusing `CurateSettings`. The filter is wrapped as a predicate function compatible with `run_predicates()` from `pipeline/quality/`.
- D-07: FineWebQualityFilter handles the "too short" and "no real sentences" audit categories. The composite predicate from `pipeline/quality/` handles the remaining categories (boilerplate, marketing) via the existing predicates.

**Export Gate (EXPORT-01)**
- D-08: `export_rag_corpus()` in `pipeline/export.py` filters on chunk `metadata_.substance_passed` instead of (or in addition to) the document-level `enriched.quality_score`. Only chunks with `substance_passed=True` are included in the gold Parquet export.
- D-09: Pre-v2.6 chunks without the `substance_passed` flag default to `True` (included) — backward compatible, no migration needed. This is forward-only per scope decision D-2.
- D-10: The export logs kept/rejected/total counts at the chunk level, extending the existing `export.rag_corpus.building` structured log.

**Eval Dataset Versioning (EXPORT-02)**
- D-11: Eval datasets generated by `pipeline/datasets.py` carry a `version` field in their metadata. After v2.6 gate improvements, existing eval sets grounded in boilerplate must be regenerated. The version tag prevents evaluating v2.6 quality against pre-v2.6 eval sets.
- D-12: The version value is derived from `filter_config_version` (PIPE-01) — a config change bumps the eval dataset version.

**Report vs Enforce Mode (QUAL-03)**
- D-13: `ChunkQualitySettings` includes `gate_mode: Literal["enforce", "report"]` with default `"enforce"`. In `report` mode, chunks are annotated (`substance_passed=False`, `rejection_reason` set) but NOT filtered from the output — they pass through to embed/index/export. In `enforce` mode, failing chunks are removed.
- D-14: Configurable via env var `KLAKE_CHUNK_QUALITY__GATE_MODE` following the project's nested settings pattern.

**Must-Not-Reject CI Fixtures (MEAS-02)**
- D-15: A `tests/fixtures/must_not_reject.yaml` file containing ~20 entries, each with `label`, `text`, and `category` (e.g., `icd_code`, `dosage`, `loinc`, `hipaa_ref`, `cardinality_constraint`). Categories map to the healthcare domain's clinical-code allowlist (Phase 19 D-08).
- D-16: A parametrized pytest test loads the YAML and asserts each entry passes both the substance gate predicates and FineWebQualityFilter at chunk scope. CI fails if any fixture is rejected.

**Filter Configuration Versioning (PIPE-01)**
- D-17: `ChunkQualitySettings` includes `filter_config_version: str` (default: `"1.0"`). The chunk stage computes a cache key `sha256(f"{parsed_artifact_id}:{filter_config_version}")`, mirroring the proven `_curation_cache_key` pattern in `curate.py:80`.
- D-18: When `filter_config_version` changes, existing chunk artifacts' cache keys no longer match, triggering re-processing on the next pipeline run. The Dagster asset checks for cache hits before running `chunk()`.

## Summary

Phase 20 wires two already-built pieces of Phase 19/17 infrastructure — `pipeline/quality/`'s pure predicate module and DataTrove's `FineWebQualityFilter` (already pinned, already used in `curate.py`) — into `chunk()`'s existing per-section persistence loop, and adds a pre-filter to `export_rag_corpus()`. Nothing new needs to be installed; `datatrove==0.9.0` is already a pinned dependency.

Three findings from direct execution against this codebase change the risk profile of this phase from "straightforward wiring" to "wiring with two load-bearing gotchas":

1. **The domain allowlist is not actually protecting anything in production yet.** Phase 19 built `DomainFilters`/`check_domain_allowlist`/`domains/healthcare/filters.yaml`, but explicitly left `DomainLoader`-based resolution out of every call site (`process.py`, `dagster_defs/assets.py`, `quality_audit.py` all pass `domain_filters=None`). Empirical testing proves this matters: `check_alpha_ratio` (already-built, Phase 19) rejects the bare string `"ICD-10 E11.9"` (alpha ratio 0.36 < 0.5 default) and `FineWebQualityFilter` rejects it too (`line_punct_ratio`, since a one-line code snippet with no terminal punctuation always scores 0). Only `check_domain_allowlist`, fed the healthcare pack's `normative_allowlists` patterns, rescues it. **Phase 20's chunk-level gate MUST wire `domain_filters` through `chunk()`, and MUST resolve it in `chunk_document`/`process_crawled` via `DomainLoader.from_name()`** (mirroring the exact pattern already used for `enrich_document`'s system prompt at `assets.py:452-455`) or the MEAS-02 must-not-reject fixtures for ICD/LOINC/dosage codes will fail in CI even though this is not literally spelled out as a numbered CONTEXT.md decision.

2. **The existing healthcare `filters.yaml` does not cover the `cardinality_constraint` fixture category.** `normative_allowlists` has patterns for `ICD-10`, `LOINC`, `RxNorm`, `§\d+\.\d+` (covers HIPAA section refs), `\d+\s*mg`, `PO\s+BID` — nothing for phrases like "2 of 4 SIRS criteria." A short cardinality phrase with exactly 6 words and no terminal punctuation fails `check_terminal_punct_ratio` (`0.00 < 0.02`) even with all other predicates passing. Either extend `filters.yaml` with a cardinality pattern, or ensure the CI fixture's cardinality-constraint sample texts end in terminal punctuation (both work; the former is more robust for real production chunks).

3. **`chunk()`'s existing WR-05 content-hash formula (`sha256(f"{parsed_artifact_id}:{text}")`) will silently defeat PIPE-01's cache-invalidation requirement unless the formula itself is changed.** Because chunking is deterministic and text doesn't change when a filter threshold changes, a re-run after bumping `filter_config_version` will hit the exact same content_hash and short-circuit through the existing `get_artifact_by_hash()` no-op branch — never re-evaluating the gate. The fix is the same one `curate.py::_curation_cache_key()` already uses for `curated_document`: fold `filter_config_version` into the hash input itself (`f"{parsed_artifact_id}:{filter_config_version}:{text}"`), so a config bump produces a *new* content_hash and therefore a genuinely new chunk artifact/gate evaluation — exactly mirroring how `curate_document`'s cache invalidation already works, just applied to a per-chunk hash instead of a per-document one.

**Primary recommendation:** Insert the gate as a filter over the `raw_chunks` list returned by `_build_token_chunks()`, before the existing per-chunk persistence loop (`chunk.py:309`). Build the predicate list as `[check_table_exemption, domain_allowlist_predicate, fineweb_predicate, check_token_floor, check_alpha_ratio, check_link_density, check_stopword_ratio, check_terminal_punct_ratio]` with exemption predicates listed first (mirroring `classify_sections()`'s established ordering discipline). Fold `filter_config_version` into the existing WR-05 hash formula rather than adding a parallel cache-check layer.

### Pattern 1: Deferred-import factory for the FineWebQualityFilter wrapper (mirrors `curate.py::_build_filters()`)

**What:** Build the `FineWebQualityFilter` instance and its `run_predicates()`-compatible wrapper in `chunk.py` (or a sibling module), not in `pipeline/quality/`. `pipeline/quality/` was deliberately kept free of DataTrove's stateful filter classes and tokenizer-invoking code paths (see Pitfall 1) — only static constants (`STOP_WORDS`, `TERMINAL_PUNCTUATION`) are imported there.

**When to use:** Any time a DataTrove filter class needs to be instantiated from chunk-scoped settings.

**Example:**
```python
# Source: mirrors curate.py:51-77 (_build_filters), verified against installed datatrove==0.9.0
from __future__ import annotations
from dataclasses import dataclass

def _build_fineweb_filter(settings: "ChunkQualitySettings"):
    """Factory returning a configured FineWebQualityFilter instance.

    Deferred import — tests that don't exercise the real filter never need
    datatrove installed at import time (mirrors curate.py's _build_filters()
    Pitfall-1 avoidance).
    """
    from datatrove.pipeline.filters.fineweb_quality_filter import (  # noqa: PLC0415
        FineWebQualityFilter,
    )
    return FineWebQualityFilter(
        line_punct_thr=settings.fineweb_line_punct_thr,
        short_line_thr=settings.fineweb_short_line_thr,
        short_line_length=settings.fineweb_short_line_length,
    )


def _fineweb_predicate(text: str, metadata: dict, *, filter_instance) -> "PredicateResult":
    """Wraps FineWebQualityFilter.filter() as a run_predicates()-compatible predicate.

    FineWebQualityFilter.filter() returns bool | tuple[bool, str] — mirrors
    curate.py::score_document()'s isinstance(outcome, tuple) handling exactly.
    """
    from datatrove.data import Document  # noqa: PLC0415
    from knowledge_lake.pipeline.quality import PredicateResult

    doc = Document(text=text, id="chunk", metadata={})
    outcome = filter_instance.filter(doc)
    if isinstance(outcome, tuple):
        passed, reason = outcome
    else:
        passed, reason = bool(outcome), "fineweb_ok"
    return PredicateResult(passed, reason or "fineweb_reject")
```
Bind chunk-scoped settings via `functools.partial(_fineweb_predicate, filter_instance=filter_instance)` before passing into `run_predicates()`'s predicate list (`run_predicates` calls every predicate as `predicate(text, metadata)` with no extra args — see `predicates.py`'s WR-03 docstring note, already established in Phase 19).

### Pattern 3: Fold config version into the WR-05 content hash (mirrors `curate.py::_curation_cache_key()`)

**What:** Change the existing hash formula in `chunk()`'s persistence loop from `sha256(f"{parsed_artifact_id}:{text}")` to `sha256(f"{parsed_artifact_id}:{filter_config_version}:{text}")`.

**When to use:** Any per-chunk content-hash computation that must be sensitive to gate configuration (PIPE-01).

**Example:**
```python
# Source: adapts curate.py:80-91 (_curation_cache_key) to chunk.py's existing
# per-chunk hash site at chunk.py:314-318
def _chunk_gate_cache_key(
    parsed_artifact_id: str, filter_config_version: str, text: str
) -> str:
    """WR-05-compatible hash that also invalidates on filter_config_version bump
    (PIPE-01). Without the version component, a re-run after a threshold change
    would hit the existing get_artifact_by_hash() no-op branch and never
    re-evaluate the gate — text is unchanged even though the gate's verdict
    should be re-computed."""
    return hashlib.sha256(
        f"{parsed_artifact_id}:{filter_config_version}:{text}".encode("utf-8")
    ).hexdigest()
```
This is a pure formula change at the existing call site (`chunk.py:317-318`) — no new cache table, no new registry method. A `filter_config_version` bump causes every chunk's `content_hash` to change, so `get_artifact_by_hash()` misses, and the full persist-with-fresh-gate-verdict path runs again, exactly mirroring how `curate_document`'s synthetic hash already forces re-curation on a `CurateSettings.filter_config_version` bump.

### Anti-Patterns to Avoid
- **Reusing `CurateSettings.filter_config_version` for chunk-level cache versioning:** CONTEXT.md D-05/D-17 are explicit that `ChunkQualitySettings` is a separate model with its own `filter_config_version` — conflating the two would mean a document-level curation threshold change silently invalidates all chunk artifacts too (and vice versa), which is not the intent.
- **Tuning `FineWebQualityFilter` thresholds looser to let short clinical codes pass directly:** Verified empirically that loosening `short_line_thr`/`line_punct_exclude_zero` to admit "ICD-10 E11.9" ALSO admits genuine nav-junk (`"Home About Contact Sitemap Search"` and `"Click here for more information..."` both pass with loosened settings — see Common Pitfalls). The correct mechanism is the domain-allowlist EXEMPTION running before FineWebQualityFilter ever evaluates the text, not threshold loosening.
- **Adding `substance_passed`/`rejection_reason` to `_RAG_CORPUS_FIELDS`:** EXPORT-01 requires *filtering* the export, not adding a new exported column. `_RAG_CORPUS_FIELDS` is a security-reviewed allow-list (T-05-08) — the gate check happens in the row-building loop as a `continue`, before the row dict is constructed, exactly like the existing `filtered_out` counter for `domain` mismatches (`export.py:296-299`).
- **Persisting rejected chunks (enforce mode) as chunk artifacts before filtering:** Run the gate over `raw_chunks` (pre-persistence, pre-S3-write) rather than filtering post-hoc after `create_chunk_artifact()` has already run — avoids polluting the registry/S3 with garbage artifacts that then need cleanup, and avoids extra S3 writes for text that will never be used.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| "Too short"/"no real sentences" detection | A custom line-ratio/punctuation heuristic | `FineWebQualityFilter` (already pinned, already proven in `curate.py`) | Production-grade FineWeb-dataset heuristics; reinventing risks re-discovering the same edge cases DataTrove already handles |
| Composite substance predicate combinator | A new ad-hoc pass/fail chain in `chunk.py` | `run_predicates()` from `pipeline/quality/` (Phase 19) | Already handles ordering discipline, exemption short-circuiting, and is unit-tested to 100% branch coverage |
| Content-addressed cache invalidation on config change | A new cache table/TTL mechanism | Fold config version into the existing WR-05 content-hash formula (mirrors `_curation_cache_key`) | Zero schema changes, reuses the proven `UNIQUE(content_hash, artifact_type)` constraint already in place |
| Clinical-code protection (ICD/LOINC/dosage) | Hardcoded regex in `chunk.py` | `check_domain_allowlist()` + `domains/healthcare/filters.yaml` (Phase 19) | Domain knowledge belongs in the domain pack (CLEAN-06); framework code stays domain-agnostic |

**Key insight:** Every piece of machinery this phase needs already exists somewhere in the codebase (Phase 17's conservation invariant, Phase 19's predicate module and domain-allowlist model, `curate.py`'s cache-versioning and DataTrove-wrapping patterns). The actual work is **wiring**, not building — but two of the wiring points (domain-filter resolution, cache-key formula) are easy to get subtly wrong in a way that makes tests pass locally while silently failing the stated acceptance criteria (see Pitfalls).

## Common Pitfalls

### Pitfall 1: Domain allowlist must be resolved and threaded through `chunk()`, or MEAS-02 fixtures fail
**What goes wrong:** Without `domain_filters` wired in, `check_alpha_ratio` rejects `"ICD-10 E11.9"` (alpha ratio 0.36 < 0.5 default — verified via direct execution) and `FineWebQualityFilter` independently rejects the same string via `line_punct_ratio` (a one-line snippet with no terminal punctuation always scores a 0% line-punctuation ratio). Both rejections happen with the CURRENT, unmodified predicate defaults.
**Why it happens:** Phase 19 built the `DomainFilters` model and `check_domain_allowlist()` predicate, but explicitly deferred `DomainLoader`-based resolution at every call site (confirmed in `19-04-SUMMARY.md`: *"Automatic DomainLoader-based resolution ... remains explicitly out of scope for Phase 19"*). CONTEXT.md's D-06/D-07 for Phase 20 describe the predicate composition but do not explicitly call out "wire `domain_filters` via `DomainLoader`" as a numbered decision — it's implied by MEAS-02's acceptance criterion, not spelled out mechanically.
**How to avoid:** In both `chunk_document` (Dagster asset) and `process_crawled` (CLI/API path), resolve `domain_filters = DomainLoader.from_name(settings.domain.domain_name).filters if settings.domain.domain_name else None` (exact pattern already proven at `assets.py:452-455` for `enrich_document`), and thread it into `chunk()`'s predicate list as `functools.partial(check_domain_allowlist, allowlist_patterns=domain_filters.normative_allowlists if domain_filters else [])`, ordered FIRST or SECOND (after `check_table_exemption`) in the predicate list so it short-circuits before any threshold predicate runs.
**Warning signs:** MEAS-02's `test_must_not_reject.py` CI test passes when called directly against the predicate function with a hand-built `DomainFilters` object, but a full end-to-end `process_crawled()`/Dagster run on a source containing bare ICD-10 codes still drops them. This is a "the test proves the mechanism works but the pipeline doesn't call the mechanism" gap — verify with an integration-level test, not just the unit-level fixture test.

### Pitfall 2: `cardinality_constraint` category is not covered by the existing healthcare `filters.yaml`
**What goes wrong:** A short cardinality-constraint sample like `"Meets 2 of 4 SIRS criteria"` (6 words, no terminal punctuation) fails `check_terminal_punct_ratio` (`0.00 < 0.02`) — verified directly. It is NOT protected by any pattern currently in `domains/healthcare/filters.yaml`'s `normative_allowlists` (`ICD-10`, `LOINC`, `RxNorm`, `§\d+\.\d+`, `\d+\s*mg`, `PO\s+BID`).
**Why it happens:** The healthcare pack's `filters.yaml` was authored in Phase 19 against CLEAN-06's four named example patterns (ICD-10, LOINC, RxNorm, dosage) — "cardinality constraint" appears as a category in MEAS-02's requirement text but was never added to the allowlist.
**How to avoid:** Either (a) extend `domains/healthcare/filters.yaml`'s `normative_allowlists` with a cardinality pattern (e.g. `\d+\s*(?:of|/)\s*\d+`), which also protects real production chunks with this shape, or (b) ensure the CI fixture's cardinality-constraint sample texts have ≥6 words AND end in terminal punctuation (naturally clears `check_terminal_punct_ratio`'s `_MIN_SAMPLE_WORDS=6` exemption boundary or the ratio check itself). Option (a) is more robust since real clinical chunks won't always be phrased as complete, punctuated sentences.
**Warning signs:** A fixture entry passes when written as a full sentence (`"Meets 2 of 4 SIRS criteria."` — with the period) but fails the moment someone "simplifies" it to the bare phrase during a later fixture edit, since nothing in the allowlist actually protects the pattern.

### Pitfall 3: The existing WR-05 content-hash formula defeats PIPE-01's cache invalidation unless changed
**What goes wrong:** `chunk()`'s existing per-chunk hash (`chunk.py:317-318`) is `sha256(f"{parsed_artifact_id}:{text}")` — it does not include any filter-config component. Because chunking is deterministic, the SAME text produces the SAME hash regardless of `filter_config_version`. A naive implementation that just adds a `filter_config_version` field to `ChunkQualitySettings` and bumps it will NOT trigger re-processing: `get_artifact_by_hash()` (`chunk.py:321`) will find the pre-existing chunk artifact (same hash) and return it via the no-op branch, without ever re-running the gate against the new config.
**Why it happens:** `curate_document`'s cache-versioning pattern (which D-17 explicitly says to mirror) works because `curate_document` has exactly ONE synthetic hash per input; chunking produces MANY chunks per input, and the existing per-chunk hash formula predates the config-versioning requirement.
**How to avoid:** Fold `filter_config_version` directly into the existing hash input: `f"{parsed_artifact_id}:{filter_config_version}:{text}"`. This is a one-line change at the existing hash computation site — no new cache table, no parallel invalidation mechanism. Verify with a test that re-chunks the same `ParsedDoc` under two different `filter_config_version` values and asserts the resulting `content_hash`/`artifact_id` values differ.
**Warning signs:** A test that bumps `filter_config_version` and re-runs `chunk()`, expecting new `substance_passed` annotations to reflect the new thresholds, silently gets back the OLD artifact's stale metadata — the test may even appear to "pass" if it doesn't specifically assert on the returned annotation values, masking the bug.

### Pitfall 4: `FineWebQualityFilter.filter()` can raise `AttributeError` depending on import order (inherited from Phase 19's finding, now load-bearing for QUAL-02)
**What goes wrong:** `FineWebQualityFilter.filter()` internally calls `datatrove.utils.text.split_into_words()`, which lazily probes for a spaCy-backed tokenizer via `importlib.metadata.distributions()`. If `importlib.metadata` has not yet been imported as a submodule anywhere in the current process, accessing `importlib.metadata` raises `AttributeError: module 'importlib' has no attribute 'metadata'` — confirmed via direct execution in this environment.
**Why it happens:** `import importlib` alone does not auto-bind the `metadata` submodule as an attribute; it must be imported (directly or transitively) at least once in-process first.
**How to avoid:** Verified empirically that this is a non-issue in practice for `chunk.py`'s actual call path: pre-importing `structlog`, `tiktoken`, `pydantic`, or `sqlalchemy` (all of which `chunk.py`/the wider pipeline import at module scope well before `chunk()` runs) already triggers `importlib.metadata` to be bound. As a defensive belt-and-suspenders measure with zero downside, add an explicit `import importlib.metadata` at the top of the module that constructs `FineWebQualityFilter` (or inside `_build_fineweb_filter()`) so the behavior is deterministic regardless of future import-order changes elsewhere in the app.
**Warning signs:** A test that imports `datatrove.pipeline.filters.fineweb_quality_filter` and calls `.filter()` in complete isolation (e.g. a fresh subprocess with minimal imports, similar to `test_quality_predicates.py`'s import-boundary test) may crash even though the same code works fine inside the full application. Do not write an isolated-subprocess crash-reproduction test as a *regression guard* without the defensive import — it will start failing spuriously as soon as another module's import order changes.

## Open Questions

1. **Should `report` mode's rejected-but-kept chunks still be embedded/indexed?**
   - What we know: D-13 says report-mode chunks are "annotated but NOT filtered from the output — they pass through to embed/index/export," and EXPORT-01/D-08 then filters them out at export time only.
   - What's unclear: Whether embedding/indexing garbage chunks in report mode (intentionally, for observability) has a meaningful cost/storage implication the planner should budget for, or whether it's negligible at current corpus scale.
   - Recommendation: Treat as intentional per D-13's explicit wording — report mode is meant to observe the gate's effect without changing pipeline behavior, cost is out of scope for this phase.

2. **Does the Dagster `chunk_document` asset need an explicit pre-check ("skip chunk() entirely if cache is warm"), or is the WR-05-formula fix (Pitfall 3) sufficient on its own?**
   - What we know: D-18 says "The Dagster asset checks for cache hits before running `chunk()`" — but `chunk()` already does its own per-chunk cache check internally (`get_artifact_by_hash()`), so re-running `chunk()` on an unchanged config is already cheap (all no-op hits, no wasted S3 writes).
   - What's unclear: Whether D-18 implies a NEW asset-level short-circuit (skip calling `chunk()` at all) purely as a performance optimization, or whether it's describing the existing per-chunk cache-check mechanism (now config-version-aware per Pitfall 3) in asset-level language.
   - Recommendation: Read D-18 as describing the existing per-chunk mechanism (now correctly config-version-sensitive) rather than requiring a new asset-level pre-check — this avoids duplicating cache logic in two places. Flag for plan-checker if the planner reads it differently.
